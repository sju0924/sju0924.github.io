# API란?
> 어떤 소프트웨어에서 다른 소프트웨어를 제어하기 위해 미리 약속된 인터페이스

>Application Program interface 의 약자

API를 사용할 때의 장점에는
* 소스 코드의 중복 제거
* 소스 코드의 표준화
* 소스 코드의 재사용성 높임
    * 개발 생산성 향상
* 소프트웨어의 내부 구조를 몰라도 사용 가능

이 있다.

# 웹 API
> 웹 프로토콜을 사용해 네트워크를 통해 호출하는 API

웹상에서 특정 `URI` 에 `HTTP` 등으로 요청을 보내고, 이에 대한 응답으로 필요한 정보를 얻어낸다.

## 웹 API와 HTTP
HTTP는 웹 서버의 HTML 데이터를 웹 브라우저가 받아 화면에 표시하기 위해 정의되었다. 시간이 지나며 HTML 기반의 정적 웹 사이트에서 js,ajax 등의 기술이 도입되면서 점점 동적인 웹 애플리케이션으로 변모되었고, 이러한 기술의 발전 덕분에 웹을 통해 구조적 데이터를 다룰 수 있게 되었다. 때맞춰 여러 대기업들이 자신이 가진 정보를 제3자가 활용할 수 있도록 `웹 API`를 공개하고 있다.<br>
![api](https://developers.naver.com/proxyapi/rawgit/naver/naver-openapi-guide/master/ko/login/overview/images/img_naverid01.png)
기업들은 웹 API를 공개하여 개발자들의 참여를 유도해, 다양한 애플리케이션에서 자사의 서비스로 연결하도록 유도해 서비스를 확대하는 성과를 거두고 있다. 최근에는 많은 인기 서비스 업체들이 자신의 서비스 API를 외부에 공개하고 있다. 이러한 기업들 중 대표적인 성공사례가 `아마존` 이다. 아마존은 광고 API를 성공시킨 바가 있었는데, 사용량이 폭주하는 자사 서비스 사이트를 안정적으로 운영하고자 서버나 스토리지 등 컴퓨팅 리소스를 API를 통해 제어할 수 있는 관리 체계를 구축하였다. 이것들이 오늘날 아마존에서 서비스하고 있는 EC2와 S3가 되었다. <br>
![s3](https://res.cloudinary.com/practicaldev/image/fetch/s--iXzorLDK--/c_imagga_scale,f_auto,fl_progressive,h_900,q_auto,w_1600/https://dev-to-uploads.s3.amazonaws.com/i/9j909fxqqhfa9x7vo8gc.jpg)
이를 웹상에 서비스로 제공하여 웹 API를 통해 사용자가 원하는 시점에 원하는 만큼의 컴퓨팅 리소스를 할당받을 수 있게 한 것을 AWS라고 이 책에서는 보고 있다. 아마존 뿐만 아니라 클라우드 컴퓨팅을 제공하는 대부분의 회사는 마소, 구글 등 인터넷 서비스를 제공하는 기업들인데, 이들이 클라우드 컴퓨팅과 웹 API 기술을 이끌고 있어 클라우드와 웹 API의 관계가 점점 공고해지고 있다. 

## 가상화 기술
웹 API를 통해 가상의 컴퓨팅 리소스를 호출하는 것은 시간을 절약하게 해 주고 편리함을 제공한다. 리소스를 즉시 이용할 수 있게 해 주고 장비를 수배하고 설치하는 시간이 들지 않는다. <br>
한편 IaaS 관점에서는 하드웨어 리소스의 자원 활용도가 높아 리소스를 효율적으로 쓸 수 있다는 면에서 가상화가 필요할 수 있지만 PaaS나 SaaS적 관점에서를 효율적인 리소스 사용보다 성능을 우선할 수도 있어 가상화가 반드시 필요하지는 않다. 따라서, *가상화 기술이 클라우드 컴퓨팅의 본질이라고는 보기 어렵다.* 클라우드 컴퓨팅에서도 성능이나 보안이 더 중요할 경우 가상화된 리소스가 아닌 물리적인 리소스를 직접 사용해야 할 수도 있다.
따라서,
> 클라우드의 본질은 리소스의 가상화 여부가 아니라 인터넷을 통해 필요한 자원을 제어할 수 있는 API 방식에 있다.

## 웹 API 의 구성 요소
웹 API는
* 인증 처리
* 제어할 대상
* 제어 행위

의 3요소로 구성된다. 이는 영문법의 문장 형식을 생각하면 쉽게 이해할 수 있다. `인증 처리`가 주어, `제어할 대상`이 목적어, `제어 행위` 가 동사와 관련된다. 주어, 동사, 목적어에 해당하는 것을 API에서 `액터`, `액션`, `리소스`라 하는데, `액터`를 식별하는 과정이 `인증 처리` 이고, `액션`은 `제어 행위`에 해당하고 웹 API를 하는 것이다. HTTP의 메소드나 헤더를 조합해서 만들어진다. `리소스`는 `제어할 대상`에 해당하고 웹 API의 URI로 표현된다. 선택적으로 보어에 해당하는 `컨디션`이 추가될 수 있는데, 이는 조건을 나타낸다. API에서 정의한 옵션이나 인증 조건 등으로 API를 호출할 때 실행 여부를 제한할 수 있다.

## 리소스
목적어에 해당하고, API 제어 대상이 되는 리소스에 대해 살펴보자. 서버, 스토리지, 네트워크 등의 클라우드 요소들을 리소스라 한다. 이 안에서도, 서버에서는 인스턴스, 이미지, 키 페어 등이 리소스이고 네트워크에서는 라우터, 서브넷, 시큐리니 그룹 등이 리소스이다. 이들은 다룬 리소스와 구분되도록 고유한 식별 키가 필요하다. 이때 사용되는 식별 키에는 두 가지 유형이 있다.
* UUID: 랜덤으로 부여된다. 인스턴스나 서브넷을 생성하면 만들어진다. 특정 리소스를 지정해야 할 때 이를 사용한다.
* 이름: 버킷 안에 들어있는 오브젝트를 지정할 때 버킷 이름과 오브젝트 이름을 사용한다.

이 중 어떤 방식이 사용되는지는 컴포넌트 단위로 결정된다. 해당 컴포넌트 문서를 읽어보면 무엇이 사용되는지 알 수 있다.<br>
리소스 중에서는 고유한 키를 가지는 속성 정보가 있는데, 이것을 `프로퍼티`라고 한다. 예를 들어 인스턴스는 가동 영역, 이미지 ID등 속성 정보가 있고, 이미지도 리소스이기 때문에 이미지 ID와 이와 관련된 인스턴스나 리소스의 정보를 갖고 있다.

## 액션
API 제어 행위에 해당한다. 액션은 리소스에 대해 행하는 동작으로, CRUD 가 대표적이다. API에서 CRUD를 지정할 때는 리소스의 URI를 호출할 때 사용되는 HTTP 메소드와 쿼리 문자열 파라미터 정보를 활용한다. 또한 조회, 수정, 삭제에는 제어 대상을 식별하기 위한 키가 필요하고, 등록은 필요하지 않다.

# 리소스와 URI

>`URI` 는 API에서 리소스를 지정할 때 사용한다.

URI는 네트워크 부분과 경로 부분으로 구성된다. 또, API를 공개하여 서비스를 제공하는 접점 혹은 API 호출을 받아 처리하는 접점을 엔드포인드라 한다. 이런 엔드포인트의 경로를 가리키기 위해 URI를 사용한다. 클라우드에서도 웹 서비스를 이용할 때와 같이 IP 주소로 직접 접근하지 않고 도메인으로 접근한다.

## 도메인
> `도메인` 은 네트워크상에서 자원의 위치를 표현할 때 사용한다. 도메인에서 점으로 구분된 단어들은 오른쪽에서 왼쪽으로 점점 하위 계층으로들어가는 계층 구조처럼 표현된다. 도메인을 오른쪽에서 왼쪽으로 볼 때 마지막 부분에 리소스가 지정된다.

> ec2-54-10-10-10.ap-northeast-2.compute.amazonaws.com

은 아마존 EC2의 외부 DNS 호스트 이름을 나타낸 것이다. .com은 TLD(Top Level Domain) 을 나타내고, 그 아래에 amazonaws, compute가 있고 그 아래에 리소스 이름을 나타내었다. 이렇게 나타낸 전체 이름을 `FQDN` 이라 한다.<br>

## DNS
> <h3>도메인과 IP 주소 사이의 변환 기능</h3>

FQDN과 IP 주소의 관계는 기본적으로 1:1 관계지만, 설정상으로 1:N, N:1 관계도 만들 수 있다. 이를 `복수 매핑` 이라 하며, 같은 도메인을 계속 사용하며 시스템 규모를 키우는데 유리한 환경을 만들 수 있다.
* 1:N 관계

는 대규모 시스템에서 활용된다. FQDN을 통해 대량의 API 요청이 들어올 시 하나의 IP 주소만으로 대응할 수 못할 수 있어, DNS에서 FQDN에 대응하는 IP 주소를 여러개 등록한다. 그러면 DNS가 순차적으로 IP 주소를 돌려쓰기 때문에 부하를 줄일 수 있다. 이를 `DNS 라운드 로빈` 이라 한다. 클라우드는 CDN 이나 로드 밸런서에서 DNS 라운드 로빈 기능을 활용하여 확장성을 높이고 있다. 또한 IP 주소 변경을 은폐하는 역할도 한다.

![roundrobin](https://post-phinf.pstatic.net/MjAxNzEwMjdfMTg2/MDAxNTA5MDYyMzA0ODcw.XgAI3xzP2ov3UkNumX33zYROPmMajcYgvFpPQZcYfq4g.9iHeDv-LfURn8pKjQ5gFEyTgeDipzs3UNKX91uee0Y0g.JPEG/mug_obj_20171027085825263.jpg?type=w1080)

* N:1 관계

는 서버 리소스를 보다 효율적으로 활용하고 싶을 때 사용한다. 이를 활용하여 `가상 호스트`를 사용할 수 있다. API를 호출하는 클라이언트에서는 스탑 리졸버라는 이름 변환 요청을 대행하는 프로그램을 통해서 캐시 DNS 서버에 도메인에 해당하는 IP 주소 정보를 물어본다. 캐시에 없아면 루트부터 최하위 도메인까지, 각 도메인의 네임스페이스를 관리하는 DNS에 되물어본다. 상위 도메인은 하위 도메인의 IP주소를 알고 있고, 하위 도메인에 상위 도메인이 보낸 IP주소가 맞나 확인하는 과정을 거친다. 이를 DNS 질의라 한다. 그런데, 이 방법을 사용하면 최상위 도메인의 DNS 서버에 상당한 부하가 걸릴 것이다. 그래서 DNS 캐시 서버를 쓰고, 여기로 질의하는 것을 `재귀적 질의`라 하고 DNS 서버 자신이 루트나 다른 DNS 서버로 질의하는 것을 `비재귀적 질의`라 한다. AWS에서 도메인을 설계할 때, 같은 리전에 있는 서비스면 같은 도메인 범위에 들어가니 비재귀적 쿼리를 많이 사용하는게 중요하다.
<br><br>
한편, 도메인 트리의 DNS 서버에 대해, 이는 누가 괸리하는 것일까? .kr은 한국인터넷정보센터가 관리하고, .com 은 기업을 의미하고 레지스트리에서 관리된다. AWS에서는, amazonaws.com 이후로는 클라우드 서비스 제공자의 도메인 범위여서, 클라우드 서비스 제공자가 운영하는 DNS 서버에서 처리한다. 그래서 외부에 공개된 클라우드 서비스로 접근하는 도메인은 사용자가 임의로 바꿀 수 없다. 경우에 따라서는 자신이 가지고 있는 도메인을 사용하고 싶을 수도 있는데, 클라우드 서비스에서는 DNS 서비스를 제공하여 이를 이용하면 된다. 대표적으로 Amazon Route 53이 있다. API를 통해 제어 가능하며 도메인 등록, 도메인-IP 주소 변환 등 서비스를 이용하여 유연한 아키텍처를 만들 수 있다.

## DNS 레코드
> <h3>DNS 서버에서 IP 주소와 도메인을 짝 지은 설정 정보</h3>

DNS 설정에는 TTL이라 하는 캐시의 수명 시간을 결정하는 정보가 있는데, DNS 레코드별로 설정할 수 있다. 직접 DNS 서버를 운영해야 한다면 BIND를 설치하고 도메인 정보를 DNS 레코드 파일에 설정하면 되고, Route 53을 이용하면 콘솔을 통해 레코드 파일을 읽거나 API로 등록한다.

## URI
><h3> 웹 API에서 리소스를 지정할 때
* URL
* URN

가 포함되어 있다.
![uri](https://media.vlpt.us/images/jch9537/post/51dcc312-8ecb-4048-80df-cbde40865e7a/image.png)
`URL` 은 네트워크상에 있는 리소스의 위치를 알려줄 때 사용한다. 클라우드에서 API 를 실행할 때 리소스 지정을 위해 URL을 사용한다. URL 은 네트워크 부분과 경로 부분으로 나누어진다. <br>
`네트워크 부분`은 프로토콜을 나타내는 스키마명,FQDN, 포트 번호로 나누어지고 그 뒤에 /로 구분되는 `경로 부분`이 있다. 이는 제어할 리소스를 식별하는 데 사용된다. 이러한 리소스들에 대한 제어 행위를 하고 싶으면 HTTP의 GET 같은 액션을 사용하면 된다. 경로 부분에는 보통 디렉터리 경로와 파일 이름이 사용되는데, 쿼리 파라미터나 프래그먼트 식별저를 추가하기도 한다. 쿼리는 조건을 지정하고 싶을 때 사용하고, 프래그먼트 식별자는 웹 페이지의 내용이 길어 전체 내용을 파악하기 어려울 때 # 문자를 사용하여 특정 위치로 이동할 수 있다.

<br>

## URN
> <h3> 네트워크와 상관없이 리소스 이름을 BNF 표기법으로 정의

`BNF` 표기법은 :나 <, >를 조합하여 구성 요소의 관계나 문법을 표현할 때 사용한다. 대표적으로 Amazon 리소스 이름이나 프로퍼티 타입이 있다. URL 이 네트워크상의 리소스를 가리키는데 사용되었다면 URN은 내부 기능에 필요한 리소스를 가리킨다. 이러한 정보들은 클라우드 내부의 정책을 제어할 때 리소스를 식별하기 위해 사용된다. <br>
> arn:aws:service:region:account-id:resource-type/resource-id 

형태로 EC2를 나타낼 수 있다.


## 엔드포인트
><h3> 클라이언트가 공개 API 를 실행하기 위해 접속하던 연결 접점
`엔드포인트`는 FQDN으로 표현되고, 일종의 게이트웨이 역할을 한다. AWS 에서는 같은 리전에 속한 VPC와 EC2를 같은 컴포넌트 법주에 있다고 간주해 같은 엔드포인트 지점을 사용한다. 여기서도 대외로 서비스할 경우 도메인으로 접속하게 된다. 
### 리소스 지정 방법
엔드포인트 URL 로만은 리소스를 제대로 지정하지 못한다. 이를 식별하기 위해
* REST API
* 쿼리 API
방식을 사용해야 한다.
`REST API` 방식은 리소스의 관계 정보를 URL 경로에 계층 형태로 표현하는 방법이다. URL의 경로 부분에서 /로 구분하여 사용하면 된다.<br>
`쿼리 API`는 액션 API를 호출할 때 옵션을 추가로 줄 수 있다. ?를 이용하여호출하고 &로 연결할 수 있다.

# 액션과 HTTP
## HTTP,쿠키, 킵얼라이브
웹 API 는 HTTP/HTTPS 프로토콜을 사용한다. HTTP는 통신할 때 상태를 유지하지 않는데, 이를 `스테이트리스`라 한다. 웹 API 역시 상태 정보를 유지할 수 없고 간단한 처리만 할 수 있다. HTTP 통신에서는 부하 분산을 위해 DNS 라운드 로빈 기법을 사용한다. 그래서 클라이언트가 이전에 접속했던 서버에 다시 접속한다고 보장할 수 없어서, 클라이언트가 이전에 요청을 보냈던 서버에 지속적으로 연결하게 만들려면 HTTP의 쿠키나 킵얼라이브라는 기술을 사용해야 한다. <br>
`쿠키`는 HTTP 통신을 하는 클라이언트나 브라우저에 서버가 보낸 정보를 저장하게 하는 기법으로, 저장된 정보를 `쿠키`라고 한다. 접속한 적 있는 웹 서버로 다시 접속하게 해주거나 클라이언트 식별이나 상태를 확인하는 용도로 사용된다. 그리고 `킵얼라이브` 는 한번 연결된 TCP 접속을 유지할 때 사용된다. 웹 API 가 연속적으로 HTTP로 통신을 하더라도 접속을 맺고 끊는 과정이 필요없어져 부담을 많이 줄일 수 있다. 이들은 HTTP 헤더를 통해 설정할 수 있다.

## HTTP 요청

HTTP 요청은
* 요청 행
* 요청 헤더
* 메시지 바디

의 세 부분으로 구성된다.

* 요청 행

> 메소드, 수신지 URI, HTTP 버전 정보 기재

* 요청 헤더
> 쿠키나 킵얼라이브같은 통신 제어 정보나 메타 데이터

* 메시지 바디
> 실제로 송수신하려는 데이터 부분. 웹 API는 주로 쿼리 파라미터로 정보 전달

## HTTP 응답

HTTP 요청을 보내면 HTTP 응답이 돌아온다. HTTP 응답에는

* 상태 행
* 응답 헤더
* 메시지 바디

로 구성된다.

* 상태 행
> HTTP 요청에 대한 결과가 정상이지 비정상인지 표현하는 3자리 숫자

* 응답 헤더
> 웹 서버가 클라이언트에게 서버 측의 정보를 전달하기 위해 사용

* 메시지 바디
> HTTP 요청에 대한 결과 데이터가 들어갑니다,

## HTTP 메소드

* GET

리소스 정보를 조회할 때 사용된다. 사용 빈도가 높다. 식별 키가 필요하다. 결과로 돌아오는 HTTP 응답에 질의 대상 리소스에 대한 정보들이 포함되어 있다. 리소스 정보를 변경하지 않기 때문에 서버에 가해지는 부하를 분산시키기가 비교적 쉽다.

* POST

리소스를 신규 생성할때 사용한다. 식별 키가 필요하지 않아 조작 대상 리소스보다 한 단계 상위 리소스 타입이 URI에 사용된다.

* PUT

리소스 정보를 수정할 때 사용하는 메소드. 식별 키가 필요하다. POST와 헷갈릴 수 있는데, POST는 식별 키 없이 새로운 리소스를 만드는 것이고 PUT은 이미 식별 키를 가지고 있는 기존의 리소스를 수정하는 것이다.

* DELETE

리소스를 삭제할 때 사용. 식별 키가 필요하다.

* HEAD

메타 데이터를 조회할 때 사용한다.

## HTTP 헤더
> HTTP 통신에 필요한 부가 정보 전달. 

쿠키나 킵얼라이브와 같은 기능이 대표적이다 HTTP 헤더는 크게 세 가지로 나눌 수 있다.

* 공통 항목
* 비표준 항목
* 클라우드 서비스에서 독자적으로 추가한 것(확장 헤더)

이 중 의미있는 헤더들은

* 표준
    * Host
    * Accept
    * Last-modify
    * If-계열
    * Authorization
    * Rage
    * Etag
    * 
* 비표준
    * Cookie

가 있다.

## HTTP 상태 코드
HTTP 요청에 대한 결과를 나타내고, 3자리 숫자로 표현된다.
* 200번대: 정상
* 300번대: 리다이렉트
* 400번대: 클라이언트 측 오류
* 500번대: 서버 측 오류

를 나타낸다.

## SOAP, REST

웹 API를 크게 두 종류로 구분한다. 메시지나 제어와 관련된 부분에서 차이를 보이는데, `SOAP`는 복잡한 비즈니스 로직을 웹 서비스 형태로 제공할때 사용한다. 엔벨로프라는 봉투와 같은 구조 안에 헤더, 바디가 XML 형식으로 만들어진다. `REST` 는 CRUD 조작처럼 간단한 제어를 할 때 사용한다.<br>
전자상거래같이 복잡한 데이터 연계가 필요한 시스템에는 SOAP 사용이 적절하지만, 클라우드 분야에서는 리소스 CRUD와 같은 단순한 처리가 대부분이다. 그래서, 이 책에서는 REST 를 중심으로 살펴볼 것이라 한다.

# ROA

## REST 기반 서비스
>ROA, 즉 리소스 지향 아키텍처란 REST API 의 사상을 기반으로 리소스 중심적인 API를 사용하는 아키텍처이다.

REST 기반 서비스에는 네 가지 설계 지침이 있다.

* 상태를 가지지 않도록 만든다.
* URI는 디렉토리 구조처럼 계층적으로 만든다.
* HTTP 메서드를 명시적으로 사용한다.
* 응답할 때는 XML 이나 JSON을 사용한다.

이 내용만 보면 그리 중요해 보이지 않지만, API가 행위 중심이 아닌 자원 중심으로 설계할 때 반드시 필요한 고려사항이다.

* 상태를 가지지 않아 구현이 쉽고 캐시를 사용할 수 있다.
* URI의 가독성이 좋고 리소스의 구조를 이해하기 쉽다
* 리소스의 상태 변화를 HTTP 메서드를 활용해 리소스 중심으로 처리하고 별도 메소드를 사용해서 행위 중심으로 처리하지 않도록 한다.
* 데이터 표현을 정규화한다.

REST API 특징 중 분산 환경 제어시 사용자가 이해해야 하는 세 가지 개념이 있다.

* 비동기
* 멱등성
* 재시도

`비동기` 는 , AWS 와 같이 엔드포인트가 공개되어 있는 경우 응답이 지연될 수 있어 먼저 보낸 요청이 뒤에 보낸 요청보다 먼저 도착한다고 보장할 수 없고, 소스가 여러 곳에 분산되어 있기도 해 처리 과정도 비동기로 처리될 수 있다는 것을 인지해야 한다.<br>
`멱등성`은 상태가 없어 같은 API를 호출하면 리소스에 변경이 발생하지 않는 한 같은 결과가 나오는 것이다. 멱등성이 보장되면 네트워크 상태 때문에 오류가 발생하더라도 HTTP 에러 코드를 보고 다시 한번 시도할 수 있다.<br>
`재시도` 결과 응답으로 오는 메타 데이터가 멱등성에 의해 같은 값이 나와 최종 정합성도 보장된다.

## API 이력 정보 확인하기
API가 어떤 처리를 했는지 실행한 이력이 보고 싶을 때는 기본적으로 클라우드가 제공하는 이력 관리 기능을 활용할 수 있다. 이를 위해 AWS에서는 API 호출 이력을 관리하는 AWS CloudTrail과 리소스 구성 형태와 설정 변경 이력을 관리하는 AWS config라는 서비스가 있다.
